# Mozart > Prototypes > Prototype1

```Message```

Η βασική υλοποίηση με τα απολύτως απαραίτητα components έγινε με Spring, Kafka, ZooKeeper και Kafka Streams.  
Μέρος της υλοποίησης είναι η δήλωση των μηνυμάτων καθώς επίσης 
και του τρόπου που γίνεται serialization & deserialization για τη μεταφορά τους προς και από τον broker.  
Ενώ φαίνεται να υπάρχει καλή υποστήριξη για serialization/deserialization δεν είναι αρκετό.  
Το JSON/string/bytes είναι βολικό αλλά δεν αρκεί από την άποψη ότι χρειάζεται 
να υλοποιήσουμε εμείς τους mappers που μετατρέπουν το JSON σε specific class.  
Μπορεί να φαίνεται αρκετό αλλά καθώς το σύστημα εξελίσσεται και προκύπτουν νέες ανάγκες σε data 
που αναπαριστούν events χρειαζόμαστε ευελιξία, δυναμικότητα, πολυμορφισμό, σημασιολογία, επαλήθευση, 
μετατροπή, serialization, deserialization, persistence, comparison, transformation, conversion, validation, "inducing", κτλ.  
Εναλλακτική είναι το Avro. Προέρχεται από το οικοσύστημα Hadoop, δουλεύει πολύ καλά με Kafka, είναι battle-tested και πολύ γρήγορο. 
Όμως και πάλι είτε θα πρέπει να δεσμευτούμε στη χρήση GenericRecord είτε στη χρήση SpecificRecord 
που απαιτεί αλλαγή στον κώδικα μιας και νέες κλάσεις πρέπει να δηιουργούνται, αλλάζουν, αφαιρούνται.  
Μπορούμε να δεσμευτούμε στη χρήση GenericRecord / JSON αλλά αργά ή γρήγορα η ανάγκη για ένα νέο layer 
που απαιτεί "μετάφραση" ενός record / JSON object σε κάτι με διαφορετική σημασιολογία θα προκύψει.

Drivers:
- Δημιουργία δυναμικών, πολυμορφικών οντοτήτων σε runtime χωρίς re-compilation
- Convention over configuration
- Διαφορετικές αναπαραστάσεις του ίδιου message
- Δημιουργία message από διαφορετικά δεδομένα. 
Π.χ. μία συσκευή στέλνει epoch millis άλλη στέλνει UTC Unix timestamp σε character sequence (String).  
- Message με σημασιολογία κοινή σε όλο το κατανεμημένο σύστημα
- Persistence Model, Domain Model, Data Exchange Model, Presentation Model, Port Model που όλα προέρχονται/προκύπτουν από ίδια ρίζα
- Serialization/Deserialization
- IDL, scripting
- Μορφή κοντά σε JSON, XML, MessagePack, etc.
- Πιθανές αλλαγές στο processing να μην επηρεάζουν αλλαγές στα δεδομένα και το αντίστροφο.
Π.χ. ανταλλάσω Temperature version 1 και ξαφνικά λόγω αλλαγή στο πεδίο πρέπει να προσθέσω νέο field.
Απαιτείται αλλαγή στον κώδικα, re-compilation, re-deployment, etc 
και αν δεν το κάνω προσεκτικά και υπάρχουν μηνύματα παλαίοτερης έκδοσης στο ίδιο topic που δεν έχουν καταναλωθεί δε θα δουλέψει τίποτα...
- Integrations (protocols, libraries, technologies, standards, etc.) 
ΧΩΡΙΣ να επηρεάζεται το κυρίως μέρος (δηλαδή ο κώδικα που γράφουμε, που ελέγχουμε και που μπορούμε να τροποποιήσουμε)

Λύσεις:
- bytes/String - JSON - Object Mapper
- bytes/String - Avro - Registry - GenericRecord

@DomainModel
class Temperature {

@Payload
class TemperatureUpdateEvent implements Serializable, Cloneable {

  private static final long serialVersionUID = 1L;
  public static final long VERSION = 1L;

  private String deviceId;
  private Double temperature;
  private String unit;
  private Long timestamp; // Epoch millis.

  // [...] Constructors, Getters, Setters [...]
}

@PersistenceModel
class TemperatureRecord implements Serializable {

  private static final long serialVersionUID = 1L;
  public static final long VERSION = 1L;
  
  @Id
  private String id;

  @Indexed
  @Field("deviceId")
  @NonNull
  private String deviceId;

  @Field("temperature")
  @NonNull
  @Min(-250)
  @Max(250)
  private Double temperature;
}
